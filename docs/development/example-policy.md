# 示例工程规范

## 示例职责

`examples/admin` 是本地开发入口，也是 `create-vela-admin` 模板的唯一长期维护源码。它应该展示 Vela Admin 的框架能力，而不是承载不可复用的复杂业务系统。

示例工程应覆盖：

1. 登录和 mock 权限。
2. 控制台。
3. 用户列表、用户详情、角色管理。
4. 多级菜单。
5. 权限按钮。
6. 403、404、500。
7. 设置中心和主题能力。

## 组件使用限制

示例页面只能使用：

1. Varlet 组件。
2. Vela Admin 已提供的组件、服务、指令和配置能力。
3. 示例工程内少量用于组织页面的轻量组件。

如果示例需要一个通用后台能力，优先评估是否应进入 `packages/framework`。不要只在示例里手写一个看似通用但无法复用的实现。

## 模板同步规则

只长期维护：

```text
examples/admin
```

不手工维护：

```text
packages/create-vela-admin/template
```

`template` 目录由 `packages/create-vela-admin/scripts/sync-template.js` 在发布前生成，由 `postpack` 清理。

## 示例复杂度

示例页面可以模拟数据和交互，但不应引入真实后端、复杂状态库或重型业务依赖。

新增示例页面时，应同时确认：

1. 是否需要 `.meta.js` 文件。
2. 菜单标题、图标、权限码和排序是否清晰。
3. 页面是否展示真实控件状态，而不是空白占位。
4. 是否会被模板同步到用户项目。

## 视觉要求

示例页面必须遵循 `docs/varlet-admin-design-system.md`。后台页面应服务信息扫描和重复操作，不使用营销式排版，不把内容区整体做成大圆角展示卡片。
