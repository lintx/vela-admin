# 组件选型规范

## 选型顺序

framework 新增 UI 或交互组件时，按以下顺序决策：

1. 优先使用 Varlet 已有组件。
2. Varlet 组件可以通过组合、轻量封装或样式 token 满足时，优先采用该方式。
3. 如果需求具有框架复用价值，将能力沉淀为 Vela Admin 组件或服务。
4. Varlet 和 Vela Admin 都无法覆盖时，再评估公开包。
5. 只有前几种方式都不合适时，才手写 UI。

## Varlet 优先规则

新增控件前应先检查 Varlet 是否已有对应能力，例如按钮、表单、弹层、菜单、标签页、开关、滑块、上传、选择器、表格相关基础控件。

如果 Varlet 组件只缺少布局组合或少量状态管理，应在 Vela Admin 内封装组合能力，而不是复制一个外观相近的新组件。

## 手写 UI 限制

手写 UI 必须满足：

1. 明确属于框架能力，不是示例工程的一次性业务界面。
2. 遵循 `docs/varlet-admin-design-system.md`。
3. 支持浅色、深色和主题 token。
4. 支持键盘访问、可见 focus 和必要的 ARIA 语义。
5. 有对应测试或验收记录。

## 第三方包评估

引入第三方包前，计划中必须说明：

1. 使用场景。
2. Varlet 和现有依赖无法满足的原因。
3. 替代方案。
4. 许可证和维护状态。
5. 对 framework 包体积和公开 API 的影响。

除非收益明确，不为示例页面引入新的运行时 UI 依赖。

## 设计系统约束

UI 默认遵循 Varlet Admin 设计系统：

- 不做营销式 hero 作为后台主界面。
- 不使用大面积高饱和主题色铺底。
- 不把页面统一包成大圆角外层卡片。
- 避免卡片套卡片。
- 表单输入必须有标签。
- 交互状态不能只依赖颜色表达。
