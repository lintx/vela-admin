# Review 规范

本规范用于定义 Vela Admin 的人工 review 和 AI review 重点。目标是优先发现会影响长期维护、公开 API、发布质量和用户升级成本的问题。

## 需要重点 review 的改动

以下改动必须在计划和总结中显式说明 review 重点：

1. 公开 API、配置项、导出类型或 CSS variables 变化。
2. 路由、权限、菜单、标签页、设置中心、主题系统变化。
3. `examples/admin` 模板行为变化。
4. `packages/create-vela-admin` 生成器、模板同步或发布包内容变化。
5. 新增运行时依赖或升级核心依赖。
6. UI 大改、响应式布局、可访问性和主题适配变化。
7. 发布流程、打包配置、npm 包边界变化。

## Review 检查项

review 时至少检查：

1. 改动是否符合任务范围，有无顺手重构或无关格式化。
2. 是否影响用户升级、公开 API 或模板生成结果。
3. 是否需要同步文档、示例、测试或 ADR。
4. 是否遵循 Varlet 优先和 Vela Admin 设计系统。
5. 是否补充或运行了匹配风险的测试。
6. 是否存在权限绕过、输入未校验、路径写入越界或 XSS 风险。
7. 是否改变发布包内容或锁文件，且原因清晰。

## AI Review 要求

AI 做 review 时应采用问题优先格式：

1. 先列风险和问题，按严重程度排序。
2. 每条问题给出文件和行号。
3. 没发现问题时，明确说明剩余风险和未验证部分。
4. 不把风格偏好当成问题，除非违反项目规范。

## Review 结论

review 结论应落到以下之一：

1. 可接受。
2. 可接受，但有明确后续事项。
3. 需要修改后再合并。
4. 需要重新设计或补充计划。

如果 review 中出现影响长期维护的决策，应提醒是否记录到 `docs/adr/`。
