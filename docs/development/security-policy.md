# 安全开发规范

本规范用于约束 Vela Admin 的权限、输入输出、模板生成和前端安全基线。它不是公开安全披露流程，后续需要时可新增根级 `SECURITY.md`。

## 权限基线

权限相关能力应默认保守：

1. 未登录默认跳转登录或进入明确的未授权流程。
2. 无权限菜单不应出现在菜单和搜索结果中。
3. 按钮权限默认行为应避免误操作，当前默认 `remove`。
4. 权限判断语义变化必须同步文档和测试。

示例工程的 mock 权限不能被描述为生产鉴权方案。

## 前端输入输出

前端渲染应避免直接插入不可信 HTML。需要展示用户输入、路由参数、错误信息或外部数据时，应保持文本渲染或明确说明转义策略。

错误信息应避免暴露本地路径、系统环境、token、账号敏感信息或完整堆栈。

## 本地存储

示例工程可以使用 localStorage/sessionStorage 模拟登录态，但文档应明确这是示例行为。框架不应默认要求业务项目把敏感 token 持久化到不安全位置。

## CLI 和模板生成

`create-vela-admin` 写入文件时必须遵循：

1. 不写出目标目录之外。
2. 不覆盖用户已有文件，除非用户明确确认。
3. 目标目录检查和错误提示应清晰。
4. 模板同步脚本不依赖 monorepo 外部路径。

涉及路径拼接、复制目录和清理目录的改动，应重点 review。

## 依赖和构建

新增依赖或构建脚本时，应避免：

1. 安装后执行不必要的远程脚本。
2. 把开发环境秘密写入示例或模板。
3. 把本地生成物、日志或 token 打包进 npm 包。

发布前应通过 `pack:*` 检查包内容。
