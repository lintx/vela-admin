# 代码组织规范

## 目录边界

`packages/framework` 只放框架运行能力、公共 API、样式和对应测试。框架包发布后应能被用户项目直接依赖，不应依赖 monorepo 私有路径。

`examples/admin` 是示例工程和模板源码。示例用于展示框架能力，不承担框架核心逻辑。

`packages/create-vela-admin` 只放项目生成器、模板同步脚本和生成器测试。`template` 目录由发布流程临时生成，不作为长期源码维护。

`docs` 放公开文档。维护规范放在 `docs/development/`，重要架构取舍后续可放在 `docs/adr/`。

## 文件拆分阈值

新增或大改文件遵循以下阈值：

| 类型 | 检查阈值 | 默认拆分阈值 |
| --- | ---: | ---: |
| Vue SFC | 300 行 | 400 行 |
| JS/TS 服务或工具文件 | 250 行 | 350 行 |
| CSS 文件 | 250 行 | 350 行 |
| 测试文件 | 350 行 | 500 行 |

达到检查阈值时，应在计划中说明是否拆分。达到默认拆分阈值时，默认应拆分，除非该文件高度内聚且拆分会明显降低可读性。

存量大文件不要求无关重构。后续如果修改触及核心逻辑，应优先考虑局部拆分，例如抽出 composable、纯函数、常量、类型、子组件或测试夹具。

## 公共 API 边界

`packages/framework/src/index.ts` 是框架公开出口。新增导出前应确认：

1. 是否属于用户需要稳定依赖的能力。
2. 命名是否和现有 API 风格一致。
3. 是否需要同步更新 README 或 `docs/` 用户文档。
4. 是否会扩大未来兼容成本。

内部实现优先保持在对应模块目录中，不为了短期复用过早公开。

## 依赖边界

新增依赖前必须说明：

1. Varlet 或现有依赖为什么无法满足。
2. 许可证是否适合公开 npm 包。
3. 包体积、维护状态和 API 稳定性。
4. 是否会进入 `vela-admin` 运行时依赖。

示例工程不应引入只为装饰或一次性页面服务的重量依赖。
