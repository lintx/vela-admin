# AI 协作规范

## 适用范围

本文件用于约束 AI 助手在 Vela Admin 仓库内的工作方式。人类贡献者可参考，但项目根级 `AGENTS.md` 是 AI 必读入口。

## 开始任务前

AI 助手应先确认：

1. 当前任务是否需要修改文件。
2. 修改是否只发生在项目目录内。
3. 是否需要读取 `docs/development/` 中的具体规范。
4. 是否需要运行命令、安装依赖、启动服务或使用浏览器。

涉及写操作时，先说明计划、影响范围、风险和验收方式，等待确认。

除只读检查、解释代码、单处错别字或链接修正等极小任务外，AI 助手进行实际开发时默认遵循 [AI 沙盒开发流程](ai-sandbox-workflow.md)：先确认主目录干净，以一个对话任务、一个确认计划或用户指定基点作为验收批次，再在 `.ai-dev/` 独立 clone 中开发、自动 checkpoint 和验证，最后只把候选 patch 应用回主目录供用户审核。

该沙盒要求适用于所有非极小写操作，包括源码、文档、配置、Git 跟踪文件、未跟踪文件和被忽略的 `.task-plans/` 文件。文件是否进入 git 跟踪不影响沙盒要求。

用户确认方案、范围或执行计划，不等于允许跳过沙盒流程。只有用户明确说“允许直接在主目录修改”或同等含义时，AI 助手才可以跳过 `.ai-dev/`；跳过前仍应说明风险、影响范围和验收方式。

开始非极小写操作前，AI 助手必须完成以下门槛：

1. 读取本文件、`AGENTS.md` 和本次任务相关的 `docs/development/` 规范。
2. 检查主目录 `git status --short`，确认没有未处理改动。
3. 确认本轮验收批次基点，并对齐 `.ai-dev/` 沙盒。
4. 在会话 todo/checklist 中明确标记“沙盒修改、checkpoint、patch 回主目录审核与验收”步骤。
5. 在完成上述步骤前，不得调用写文件工具、删除文件、移动文件或执行会改变主目录内容的命令。

## 计划管理

除错别字、单处链接修正、只读检查等很小修改外，AI 助手必须先制定计划，再按计划执行。

计划分为两类：

1. 长期计划：影响路线图、架构、规范、发布或跨任务维护，应写入 `docs/plans/` 并加入 git 跟踪。
2. 短期计划：只服务当前任务执行，应放入 `.task-plans/`，不纳入 git 跟踪，任务结束后删除。

执行计划时必须使用可更新的 todo/checklist 机制标记状态：

1. 同一时间只允许一个事项处于进行中。
2. 完成一个阶段后立即更新状态。
3. 中断或恢复任务时，先读取计划和状态，再继续执行。
4. 不依赖上下文记忆判断当前执行到哪一步。

会话内 todo/checklist 只能记录当前交互进度，不能替代 `.task-plans/` 短期计划文件或 `docs/plans/` 长期计划文件。

计划文件关闭必须经过维护者确认：

1. 短期计划在任务完成并经维护者确认关闭后，才可以删除。
2. 长期计划完成后，应先把结论合并到稳定文档、路线图或 ADR；是否删除、标记完成后保留，或继续追踪，必须由维护者确认。
3. 流程偏差类计划不得由 AI 助手单方面判断为完成并清理，应先说明偏差、影响和沉淀建议。

## 成本控制

AI 助手应把同一对话中的上下文、已验证结果和用户确认过的设计视为可复用信息，避免把每个小反馈都当成全新任务从头执行。

### 调查和试错

1. 遇到 UI 或浏览器问题时，先确认当前服务端口对应的工作区、页面路径、视口尺寸和 DOM 结构，再编写自动化检查脚本。
2. 不凭记忆猜选择器。已有源码可读时，先读源码；页面状态不确定时，先做一次小范围 DOM 快照。
3. 同一类浏览器验收应复用脚本片段，例如登录态注入、设置项 localStorage、桌面视口和主题生成器打开流程。
4. 如果发现服务未热更新或端口对应旧工作区，应先记录端口来源，再决定是否启动新服务；不得在多个端口间反复试错。
5. 在 `.ai-dev/` 中修改文件时，如果 IDE 工具不能确认项目根目录，应使用显式 `.ai-dev/...` 路径的 patch 或命令，避免误改主目录。

### 验收分级

AI 助手应根据改动风险选择验证范围：

1. 只改文档：检查 Markdown 链接和 `git diff --stat`，不运行测试和构建。
2. 只改文案、按钮类型、局部样式：优先运行相关定向测试和一次浏览器复查。
3. 改公开 API、导出入口、构建边界或共享组件：运行 framework 全量测试和 framework build。
4. 改示例入口、主题、布局或用户可见 UI：在必要测试通过后运行 example build，并做浏览器验收。
5. 同一对话中已在当前基点通过的全量验证，不因纯文案或等价样式微调重复执行；应说明复用依据和本次补充验证。

### 连续反馈批次

当维护者连续反馈同一块 UI 的视觉问题时，AI 助手应把这些反馈合并为一个视觉调整批次：

1. 先列出本批次要修的具体点。
2. 一次性修改相关样式和测试。
3. 用一次定向测试和一次浏览器复查验证本批次。
4. 只有触及公开 API、构建边界或跨模块逻辑时，才升级为全量验证。

## 执行原则

1. 使用中文说明计划、进度、问题和总结。
2. 优先读取现有代码和文档，再提出方案。
3. 最小化修改，不改无关文件。
4. 不主动提交 commit，不创建分支，不改写历史。
5. 不覆盖用户已有改动。
6. 不留下无关临时文件、生成物或格式化噪音。

## UI 任务流程

涉及 UI 时，AI 助手应：

1. 先读 `docs/varlet-admin-design-system.md`。
2. 先判断 Varlet 是否已有组件。
3. 说明使用 Varlet、Vela Admin 封装、第三方包或手写 UI 的理由。
4. 完成后按 `testing-and-verification.md` 做对应验收。

## 文档任务流程

涉及文档时，AI 助手应：

1. 判断文档面向用户还是维护者。
2. 放到正确目录。
3. 更新相关入口链接。
4. 避免复制已有规则造成漂移。

## 决策沉淀

开发过程中如果出现影响长期维护的决策，AI 助手应提醒用户是否记录到 `docs/adr/`。

常见触发场景包括：

1. 新增或改变公开 API。
2. 引入运行时依赖。
3. 调整目录结构、发布边界或模板同步机制。
4. 制定兼容、弃用或迁移策略。
5. 改变 UI 设计系统、主题机制或组件选型原则。

## 结束任务前

最终回复应包含：

1. 实际修改的文件。
2. 已运行的验证命令。
3. 未能验证的部分和原因。
4. 后续建议，仅在确实有价值时提出。

如果本次任务不是很小修改，并且触发了维护规范，AI 助手应提醒维护者是否需要记录到 `docs/plans/maintenance-governance-trial.md`。未得到确认前，不主动写入试运行记录。
