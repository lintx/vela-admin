# 文档规范

## 文档分类

`docs/` 根级文档面向使用者：

- 快速开始。
- 配置参考。
- 路由、权限、主题。
- 架构说明。
- 设计系统。
- 路线图。
- 验收记录。

`docs/development/` 面向维护者和 AI：

- 代码组织。
- 组件选型。
- 文档存放和切分。
- 示例工程约束。
- 测试与验收。
- AI 协作流程。

后续如果出现重要架构取舍，优先新增 `docs/adr/NNNN-title.md`，而不是把决策过程塞进 README。

`docs/adr/` 用于记录影响长期维护的决策。不是每个小判断都需要 ADR，但公开 API、依赖引入、目录结构、兼容策略、弃用策略、发布流程和架构边界发生变化时，应提醒维护者是否需要记录。

计划文档按生命周期存放：

1. 长期计划进入 `docs/plans/` 并加入 git 跟踪。
2. 短期执行计划放入 `.task-plans/`，不纳入 git 跟踪，任务结束后删除。
3. 需要持续追踪的计划应使用 checklist 标记状态，避免只依赖上下文记忆。
4. 长期计划完成后必须收尾：过程性计划默认删除，先把结论合并到稳定文档或 ADR；确有长期查阅价值时才保留并标记完成。

## 切分规则

单篇文档超过 300 行时，应检查是否能按主题拆分。超过 500 行时，默认应拆分或增加目录索引。

用户文档按功能切分，维护者文档按职责切分。不要把面向用户的使用说明和面向维护者的执行规则混在同一篇文档里。

## README 职责

README 是使用者入口，不是完整手册，也不是维护者手册。README 应保留：

1. 项目定位。
2. 面向使用者的设计取向。
3. 快速开始。
4. 基础用法。
5. 已提供能力和不强制绑定的内容。
6. 用户文档索引。
7. 维护者文档入口链接。

仓库结构、模板维护、测试验收、打包发布和维护原则应放入 `docs/development/`，避免 README 从用户入口膨胀成维护者手册。

## 写作规则

1. 使用中文描述维护规则。
2. 每篇文档开头说明目标和适用范围。
3. 规则尽量写成可执行的“必须、应、禁止、默认”。
4. 避免只写抽象原则，关键规则需要给出路径、命令或判断标准。
5. 文档改动涉及行为变化时，应同步更新相关入口链接。

## 链接维护

新增或移动 Markdown 文档后，应检查：

1. README 是否需要新增入口。
2. `docs/development/README.md` 是否需要新增入口。
3. 相对链接是否仍然有效。
4. 是否出现两处内容重复且可能漂移。
