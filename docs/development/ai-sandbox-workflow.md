# AI 沙盒开发流程

## 目标与适用范围

本文件约束 AI 助手在 Vela Admin 仓库内进行实际开发时的隔离流程。目标是同时满足：

1. 主目录作为用户审核区，只接收沙盒验证后的候选 patch。
2. AI 修改过程必须自动 checkpoint，避免误删、误改后无法恢复。
3. AI 试错产生的临时提交不污染主仓库的正式历史和 Git 对象库。

除只读检查、解释代码、单处错别字或链接修正等极小任务外，AI 助手默认应使用本流程。用户明确要求直接在主目录修改时，以用户即时指令为准，但仍应先说明风险。

## 目录约定

项目根目录作为正式工作区和用户审核区，只接收从 `.ai-dev/` 生成的候选 patch：

```text
<repo-root>/
```

AI 专用开发目录固定为项目根下的 `.ai-dev/`，它是当前仓库的独立 clone：

```text
<repo-root>/.ai-dev/
```

`.ai-dev/` 必须被主仓库忽略，不得提交其中任何文件。该目录长期保留，不在每轮开发结束后删除，以减少重复 clone、依赖安装和磁盘擦写。

## 验收批次

每次非极小修改都必须先确定一个验收批次。默认规则如下：

1. 有用户确认的执行计划时，一个计划对应一个验收批次。
2. 没有计划但任务边界明确时，一个对话中的一个明确任务对应一个验收批次。
3. 用户明确说“以这里为基点”或同等含义时，以该确认点作为验收批次起点。

验收批次用于决定沙盒 checkpoint、patch 生成、主目录审核和后续接受或回退的边界。不得把多个风险域混在一个验收批次里。

## 开始任务前

开始新一轮验收批次前，必须先检查主目录状态：

```sh
git status --porcelain
```

如果有任何输出，AI 助手必须停止开发，提醒用户先处理主目录改动。主目录不干净时，不得进入 `.ai-dev/` 开始新验收批次，也不得尝试自动 stash、reset、checkout 或覆盖用户改动。

主目录干净后，记录当前 `HEAD` 作为本轮基准：

```text
base = 主目录 HEAD
```

本轮候选 patch 必须从这个基准生成，不能无边界地同步 AI 目录当前文件。

## 初始化与对齐

如果 `.ai-dev/` 不存在，AI 助手应先说明创建计划、影响范围和验收方式，等待用户确认后再创建。创建时应把主目录作为本地 remote，便于后续同步主目录正式提交。

如果 `.ai-dev/` 已存在，每轮开始前必须让它和主目录 `HEAD` 对齐：

```sh
git fetch <主目录remote>
git reset --hard <主目录HEAD>
```

只有当 `.ai-dev/` 的 Git 仓库已经获取到主目录 `HEAD` 对应对象后，才能 reset 到该提交。如果主目录存在未提交改动，则不能通过 reset 对齐，应先回到“开始任务前”的规则处理。

AI 目录对齐时可以清理忽略文件，但不得清理主目录：

```sh
git clean -fdX
```

是否清理未忽略文件需要单独说明并等待确认。

## AI 目录开发规则

AI 助手只在 `.ai-dev/` 中进行实际开发和试错。开发过程中必须在每个可描述阶段自动提交 checkpoint，例如：

```text
checkpoint: 调整布局
checkpoint: 修复构建错误
checkpoint: 回退不满意的实现
```

这些 checkpoint 只用于 AI 目录内恢复状态、生成候选 patch 和定位回退点，不属于正式提交历史。除用户明确要求外，不得把这些提交 cherry-pick、merge 或 rebase 到主目录。

checkpoint commit 信息应简短描述阶段结果，例如：

```text
checkpoint: 更新沙盒验收规则
checkpoint: 修复 create 路径边界测试
```

如果一次修改尚未形成可验证阶段，但需要切换上下文，也应先创建 checkpoint 或明确说明为何暂不提交。

如果需要暂时搁置一个需求并开发另一个需求，优先在 `.ai-dev/` 内使用 Git 分支或 worktree 管理临时状态。切换任务前必须说明当前状态、基准提交和潜在冲突风险。

## patch 到主目录审核

AI 目录修改完成后，必须先确认 AI 目录工作区干净。若存在未提交改动，应提交为 checkpoint，或在用户确认后丢弃。

候选变更只通过 diff 或 patch 回到主目录：

```sh
git diff <base>..<target-checkpoint>
```

回到主目录应用 patch 前，必须再次检查主目录：

```sh
git status --porcelain
```

如果主目录不干净，必须停止并等待用户处理。不得在主目录有未处理改动时强行应用 AI patch。

应用 patch 后，主目录进入本验收批次的审核状态。AI 助手必须说明 patch 来源、目标 checkpoint、影响文件和验证结果；主目录仍由用户决定是否正式提交。AI 助手不得主动提交 commit。

如果用户要求继续调整同一验收批次，AI 助手应继续在 `.ai-dev/` 基于当前 checkpoint 修改并再次 checkpoint，再生成增量或完整候选 patch。不得在主目录直接补改。

## 验收后决策

用户验收主目录审核区变更后，可以选择：

1. 接受：保留主目录变更，进入下一批或等待用户自行提交。
2. 调整：回到 `.ai-dev/` 继续修改，重新 checkpoint 和 patch 到主目录审核。
3. 回退：使用本验收批次 patch 的 reverse patch 回退主目录变更。

回退不得使用 `git reset --hard`、`git checkout --` 或其它会覆盖用户改动的破坏性命令，除非用户明确要求并确认风险。执行回退前必须再次检查主目录状态，确认只有本验收批次相关待审核变更；如果混入其它改动，必须停止并让用户决策。

## 冲突与失败处理

如果 patch 无法干净应用，AI 助手应报告冲突文件和原因，并在主目录基于真实状态处理冲突。不得偷偷覆盖主目录文件，也不得把 AI 目录的完整文件树复制回主目录。

如果本轮开发失败，AI 助手应说明可选 checkpoint 和失败原因。用户可以选择：

1. 丢弃本轮 AI 目录改动，并在下一轮重新对齐主目录。
2. 选择某个 checkpoint 生成 diff，回主目录继续。
3. 暂时保留 `.ai-dev/` 当前状态，后续继续分析。

## 验收要求

AI 目录内通过测试只说明候选实现可用。最终 diff 应用到主目录后，必须按 `testing-and-verification.md` 重新运行与修改范围匹配的验证命令。

只改文档时，至少检查：

```sh
git diff --stat
git diff --check
```

修改 framework、example、create 包、发布边界、依赖或安全相关内容时，按对应维护规范选择更完整的验证命令。

## 禁止事项

AI 助手不得：

1. 在主目录不干净时开始 AI 沙盒开发。
2. 自动 stash、reset、checkout、删除或覆盖主目录中的用户改动。
3. 把 `.ai-dev/` 中的 checkpoint 当作正式历史同步到主目录。
4. 在主目录直接补改本应属于验收批次的候选变更。
5. 通过复制整个 `.ai-dev/` 文件树覆盖主目录。
6. 在未说明影响范围并获得确认前创建、删除或重建 `.ai-dev/`。
7. 把 `.ai-dev/` 中的临时文件、构建产物或依赖目录提交到主仓库。
